# Flutter 前后端联调：网络层搭建实施方案

## 目标

将 Flutter App 从**纯 Mock UI 原型**升级为**真正连接后端 API 的可运行应用**。分阶段推进，先打通最核心的认证 + Chat 发信链路，再逐步覆盖其他模块。

---

## 现状分析

### 后端已有 API（按前端需求排列）

| 模块 | 端点 | 方法 | 认证 | 状态 |
|------|------|------|------|------|
| 认证 | `/auth/register/email` | POST | 无 | ✅ 可用 |
| 认证 | `/auth/login/email` | POST | 无 | ✅ 可用 |
| 认证 | `/auth/login/google` | POST | 无 | ✅ 可用 |

### 后端缺失的 API（前端需要但不存在）

> [!WARNING]
> 以下接口是前端列表页和详情页必需的，但后端目前**没有对应的 GET 端点**。这属于"读取侧"的空白，需要在后端补建。

| 需求 | 描述 | 建议端点 |
|------|------|----------|
| 获取当前用户信息 | 登录后拉取 profile | `GET /auth/me` |
| 获取我的 Agent 列表 | Hub 页 carousel 数据源 | `GET /agents/mine` |
| 获取 DM 会话列表 | Chat 页左侧列表 | `GET /content/dm/threads` |
| 获取 DM 消息历史 | 进入聊天详情后拉取 | `GET /content/dm/threads/:id/messages` |
| 获取 Agent 大厅列表 | Hall 瀑布流 | `GET /agents` |
| 获取 Forum 话题列表 | Forum 页 | `GET /content/forum/topics` |
| 获取话题详情+回复 | Forum 详情 | `GET /content/forum/topics/:id` |

## 分阶段实施计划

### Phase 1：基础设施层（网络 + 认证）

搭建整个 App 的网络通信骨干和认证状态管理。

#### [NEW] `app/lib/core/network/api_client.dart`
- 使用 Dart 内置 `http` 包（无需新增依赖，Flutter SDK 自带 `dart:convert` + `dart:io`）
- 封装 `ApiClient` 类：`baseUrl`、`authToken`、通用 `get/post/delete` 方法
- 自动注入 `Authorization: Bearer <token>` 头
- 统一错误处理（`ApiException`）

#### [NEW] `app/lib/core/network/api_exception.dart`
- 统一异常类型：`statusCode`、`message`、`body`

#### [NEW] `app/lib/core/auth/auth_state.dart`
- `AuthState` 模型类：`token`、`userId`、`email`、`displayName`
- 提供 `isLoggedIn` 状态判断

#### [NEW] `app/lib/core/auth/auth_repository.dart`
- `registerWithEmail(email, displayName, password)`
- `loginWithEmail(email, password)`
- `loginWithGoogle(email, displayName, providerSubject)`
- `loginWithGitHub(email, displayName, providerSubject)`
- 所有方法返回 `AuthState`（含 token）

#### [MODIFY] `app/lib/app_shell.dart`
- 在 `_AgentsChatAppShellState` 中持有 `ApiClient` 和 `AuthState` 实例
- 将 `ApiClient` 透传给需要网络请求的 Surface（通过构造函数参数）
- Hub 页的登录操作改为调用真实 API

#### [MODIFY] `app/pubspec.yaml`
- 添加 `http: ^1.2.0` 依赖

### Phase 3：Chat 模块联调（核心 DM 发信）

打通"人类通过激活的 Agent 发起私信"的完整链路。

#### [NEW] `app/lib/core/network/chat_repository.dart`
- `sendDirectMessage(recipientType, recipientId, content, activeAgentId)` → `POST /content/dm`
- `getThreads(activeAgentId)` — 需后端补 `GET /content/dm/threads`
- `getMessages(threadId)` — 需后端补 `GET /content/dm/threads/:id/messages`

#### [MODIFY] `app/lib/features/chat/chat_screen.dart`
- 底部 Composer 的发送按钮绑定 `ChatRepository.sendDirectMessage`
- 从 Hub 的 `selectedAgentId` 获取当前激活的 Agent，作为 `activeAgentId` 传入
- 进入聊天详情页时调用 `getMessages` 加载历史

#### [MODIFY] `app/lib/features/chat/chat_view_model.dart`
- 增加 `activeAgentId` 字段
- 增加 `addMessage(ChatMessageModel)` 方法用于本地乐观插入

## User Review Required

> [!IMPORTANT]
> **后端缺失读取 API 是核心阻塞点。**
> 前端联调不仅仅是 Flutter 侧的工作。后端目前只有"写入"类接口（发消息、创建话题、发起辩论），缺少几乎所有的"读取/列表"类接口。
> 
> 我的建议是：**Phase 1 + Phase 2 可以立即开始**（认证接口和发信接口已完备），同时在后端并行补建缺失的 GET 端点。
>
> 你希望我：
> 1. **先做 Phase 1 + Phase 2**（网络基建 + Hub 登录 + Chat 发信），后端读取接口后续再补？
> 2. **先补后端的 GET 接口**（`/agents/mine`, `/content/dm/threads` 等），然后一次性全通？
> 3. **两边同时推进**？

> [!IMPORTANT]
> **全局状态共享问题。**
> 当前 App 的各个 Tab（Hall/Forum/Chat/Live/Hub）之间没有共享状态机制。`selectedAgentId`（当前激活的 Agent）存在于 Hub 的 ViewModel 里，但 Chat 模块需要读取它来作为 `activeAgentId`。
>
> 我计划在 `AppShell` 层引入一个轻量级的全局状态容器（不引入 Provider/Bloc 等第三方库，用 Flutter 原生的 `InheritedWidget` 或直接通过构造函数透传），让 `selectedAgentId` 和 `AuthState` 可以被所有 Tab 共享。
>
> 如果你有偏好的状态管理方案，请告知。
