# Learnings

- 2026-03-28: Task 1 uses compile-time Flutter config through `app/tool/dart_define.example.json`, with defaults in `app/lib/core/config/app_environment.dart`, so `flutter test` does not depend on local env files.
- 2026-03-28: The Nest bootstrap validates required env vars before the app starts; missing `DATABASE_URL` and `JWT_SECRET` fail fast with an explicit error from `server/src/config/environment.ts`.
- 2026-03-28: After machine restart, Docker CLI became available and `docker compose -f server/docker-compose.yml up -d postgres redis minio` succeeded, confirming the local PostgreSQL, Redis, and MinIO bootstrap is working.
- 2026-03-28: Flutter `integration_test` files can be fully wired and still fail on `-d windows` until the app has a generated `app/windows/**` runner; the Dart test file and `integration_test` SDK dependency alone are not enough.
- 2026-03-29: TypeORM CLI should not reuse the full Nest runtime environment loader for migrations; a schema-only datasource that loads `DATABASE_URL` from `.env.local`, `.env`, or `.env.example` avoids unrelated Redis/MinIO/JWT failures during `pnpm --dir server migration:run`.
- 2026-03-29: Parallel Jest workers will race if multiple domain suites recreate the same PostgreSQL database; per-suite unique temp database names plus teardown keep `test/domain/**` stable in both targeted and full-suite runs.
- 2026-03-29: In this repo, `TypeOrmModule.forRootAsync()` must not inherit the CLI `migrations/*{.ts,.js}` glob. Under the current NodeNext/CJS runtime path, that makes Nest boot try to import raw TypeScript migrations and triggers ESM/CJS loader errors even though `pnpm migration:run` still works via `ts-node`.
- 2026-03-29: The root Jest config in `server/package.json` must match both `*.spec.ts` and `*.e2e-spec.ts`; otherwise explicit acceptance commands like `pnpm exec jest --runInBand test/auth/auth.e2e-spec.ts` report “No tests found” even when the files exist.
- 2026-03-29: The reusable Nest test-app harness must not call `jest.resetModules()` before requiring `AppModule`, because that reloads a second copy of Nest internals and breaks `TypeOrmModule` dependency resolution for `ModuleRef`.
- 2026-03-29: Editing an already-applied TypeORM migration caused live schema drift: the persistent `agents_chat` database kept the original `users` table while fresh test databases replayed the edited migration. The reliable fix was to restore the historical migration’s original `users` shape and add a new follow-up migration for `password_hash`, `provider_subject`, and `CHK_users_auth_identity_binding`.
- 2026-03-29: Task 5 federation `dm.send` happy paths must respect the existing Task 4 DM policy defaults; self-owned agents start at `approval_required`, so recipient agents need an explicit `agent_policies.dm_acceptance_mode = 'open'` (or another allowed policy path) before transport-level DM tests will succeed.
- 2026-03-29: Live Task 5 QA on the shared `agents_chat` database required running the new additive migration before booting the Nest server, because runtime TypeORM config does not auto-apply schema changes and the new `agent_connections.webhook_secret` column is read immediately by the delivery sweeper.
- 2026-03-29: In Task 6, re-saving fully loaded TypeORM entities that carry nullable relation properties (`debate_sessions`, `thread_participants`, and `assets`) can silently null foreign-key columns and trip PostgreSQL check constraints; targeted `repository.update(...)` calls are safer for state-only changes.
- 2026-03-29: Local MinIO on this host already serves the `agents-chat-local` bucket, and a lightweight SigV4 presigned `PUT` plus presigned `HEAD` flow is enough to issue uploads and verify completions without introducing new package dependencies outside Task 6 scope.
- 2026-03-30: On the 430x932 Flutter mobile golden viewport, chip text in dense glass headers must ellipsize instead of assuming infinite width; otherwise new Hall summary/status rows overflow and destabilize both runtime layout and goldens.
- 2026-03-30: For Task 10 chat in the shell-constrained mobile viewport, a compact list-first flow works reliably; trying to keep the conversation list and thread detail visible at the same time starved the detail panel and broke thread/request QA.
- 2026-03-30: Task 10 thread/request panels need tight-height fallbacks or scrolling because the app shell chrome can leave under ~220px for chat content in widget and integration tests.
- 2026-03-30: Legacy server debate fixtures can seed `debate_sessions.status = live` directly without creating the initial pending `debate_turns` row, so turn validation must resolve seat ownership first and lazily materialize turn 1 for backward-compatible seeded-live tests.
- 2026-03-30: Some older moderation debate fixtures still seed only one occupied seat; Task 11 compatibility code must avoid emitting extra follow-up lifecycle events in that legacy path or archive assertions that expect exactly the original authored turn event will drift.
- 2026-03-30: This server package had no existing UUID route-param validation pattern in controllers, so malformed debate IDs reached the database layer; a narrow `ParseUUIDPipe` on the debate read/archive GET routes is enough to convert malformed IDs into client errors while preserving normal 404 handling for valid-but-missing UUIDs.
- 2026-03-31: Task 12 Hub carousel cards also need compact vertical budgets on the 430x932 shell viewport; reducing icon padding and collapsing the secondary headline to one line eliminated PageView card overflows during `integration_test` on Windows.
- 2026-03-31: For Hub import QA, a read-only `TextFormField` can be asserted reliably through its descendant `EditableText.readOnly` in integration tests, which avoids introducing extra controller-only plumbing just for test visibility.
